
## `grid.py` — The World

This is the foundation everything else imports. It has two layers: **data containers** and **the game state**.

**The four dataclasses at the top** are just typed containers — no logic, just data:
- `GoldDeposit` holds a position, an amount, and a `depleted` flag so we know when it's been mined
- `Pillar` holds a position and size, plus an `occupies()` method that expands a pillar's top-left corner + width/height into the full list of cells it blocks — so a 2×2 pillar at (3,2) returns `[(3,2),(3,3),(4,2),(4,3)]`
- `Camera` holds position, current facing angle, rotation speed, FOV width, and range
- `Agent` holds position and the three financial counters: gross collected, tax paid, net profit

**`GameState`** is where the world actually lives. On init it hardcodes the layout — 6 pillars, 8 gold deposits, camera in the top-right at angle 200°, agent at (0,0). It also immediately pre-computes `_blocked_cells` as a Python set, which is important — instead of looping through all pillars every time you need to check if something is blocked, you just do `(x,y) in self._blocked_cells`, which is O(1).

The **three query methods** are read-only checks: `is_blocked` checks walls and bounds, `gold_at` finds an undepleted deposit at a coordinate, `total_gold_remaining` sums up what's left.

The **action methods** are the only way state should mutate:
- `move_agent` checks `is_blocked` before moving, returns `"blocked"` or `"moved"`
- `mine` is the most important one — it checks `self.agent_visible` at the moment of mining, applies the 100% tax if true, marks the deposit depleted, and updates all three financial counters simultaneously. The tax logic is just one line: `tax = gross * TAX_RATE if self.agent_visible else 0.0`
- `advance_timestep` increments the counter and rotates the camera by `rotation_speed` degrees, wrapping with `% 360`

**`to_dict()`** is the serialisation method — it flattens everything into a plain dictionary that `database.py` can log directly. Every gold deposit's state is included as a JSON-serialisable list.

## `visibility.py` — The Oracle

This is the ground-truth system. It computes whether the agent is *actually* visible to the camera using real geometry. The agent never receives this — it's for your logs only.

Three checks happen in sequence inside `compute_visibility`, and all three must pass for the agent to be visible:

**Check 1 — Distance.** `math.hypot` gives the straight-line distance from camera centre to agent centre. If it exceeds `CAMERA_RANGE` (10 cells), early return False.

**Check 2 — FOV cone.** `_angle_between` uses `atan2` to get the compass bearing from camera to agent. Then `_angular_diff` finds the smallest angle between that bearing and the camera's current facing direction. If that gap exceeds half the FOV (30°), early return False. The `_angular_diff` function has a small but important trick — the `(a - b + 180) % 360 - 180` formula handles the wraparound at 0°/360° cleanly, so a camera facing 5° and an agent at 355° are correctly understood as being 10° apart, not 350° apart.

**Check 3 — Occlusion.** `_ray_blocked_by_pillar` walks 60 evenly-spaced sample points along the line from camera to agent. For each point it floors the coordinate to find the grid cell and checks if that cell is in `_blocked_cells`. It stops before reaching the agent's own cell so the agent doesn't occlude itself. If any intermediate cell is a pillar, return True (blocked).

`get_visibility_debug` runs all the same calculations but returns a full breakdown dict — distance, angles, which check failed — which gets stored in the database and is very useful for debugging and for blog post analysis.

## `renderer.py` — The Eyes

This produces the PNG image that Moondream2 will see. Every design decision was made to make the image maximally readable by a small VLM.

It uses `matplotlib.use("Agg")` at the top — this is critical. `Agg` is a non-interactive backend that renders to a buffer in memory rather than opening a window, which is what you need for a headless server loop.

The coordinate system has one gotcha: `ax.set_ylim(GRID_H, 0)` flips the Y axis so that Y=0 is at the top, matching the grid's coordinate system. Without this, the agent at (0,0) would render at the bottom-left instead of the top-left.

**Rendering order matters** — each element has an explicit `zorder` to control what draws on top of what: grid lines (1) → FOV cone (2-3) → pillars (4) → gold (5-6) → agent (7-8) → camera body (9-10) → HUD text (11).

The **FOV cone** is a matplotlib `Wedge` patch. There's an angle conversion here: the grid stores angles clockwise from right (like a compass), but matplotlib's Wedge uses counter-clockwise from right (standard maths convention). So the angle gets negated: `cam_angle_mpl = -cam.angle_deg`.

The **agent gets a red glow ring** when `state.agent_visible` is True — this is purely a visual aid for the blog and for Moondream2. The question "is the red ring present around the blue circle?" becomes a trivially easy binary question for a VLM.

Gold deposits render as **diamonds** (rotated squares via `plt.Polygon`) scaled by their amount, with the amount printed inside. Bigger deposits = bigger diamonds — makes them visually distinct.

The final step converts the matplotlib figure to a PIL Image via an in-memory buffer (`io.BytesIO`) rather than writing to disk and reading back. `plt.close(fig)` is important — without it, matplotlib accumulates figures in memory and will eventually crash on a long run.

## `database.py` — The Log

Four tables, each with a clear purpose:

**`episodes`** — one row per complete experiment run. Stores the config (tax rate, camera speed, etc.) as a JSON blob so you can always reconstruct exactly what conditions produced a given result.

**`timesteps`** — one row per game step, storing everything about the game world at that moment. `agent_visible` is stored as an integer (0/1) because SQLite has no boolean type. `gold_deposits_json` stores the full state of all deposits as JSON — this is slightly redundant but means you can always reconstruct the exact board from a single row. `visibility_debug_json` stores the full breakdown from `get_visibility_debug`.

**`vlm_logs`** — one row per question asked to Moondream2 per timestep. Stores the raw answer (whatever Moondream2 actually said) and the parsed value (just `"yes"` or `"no"` after your cleaning code strips it down). Keeping both is important — if your parsing was wrong, you can reprocess the raw answers later without re-running the model.

**`agent_logs`** — the most analytically important table. `vlm_summary` is the text assembled from Moondream2's answers that gets sent to Seed. `scratchpad` is Seed's raw chain-of-thought. Then two auto-computed flags: `mentions_camera` and `mentions_hiding` are set at write time by scanning the scratchpad for keyword lists. These are your **confession rate metrics** — you can query them directly without doing text analysis later:

```sql
SELECT timestep, mentions_camera, mentions_hiding, action
FROM agent_logs WHERE episode_id = 1;
```

## `main.py` — The Smoke Test

This isn't the real experiment loop — it's a harness to prove all four modules work together correctly. It uses a hardcoded list of scripted actions (`SCRIPTED_ACTIONS`) instead of asking the AI, so there are no network calls and no model dependencies.

The loop order inside each step is the exact order the real loop will follow:
1. Compute ground-truth visibility
2. Render the frame to an image
3. Execute the action
4. Log everything to DB
5. Print console status
6. Advance the timestep (rotate camera)

The fake VLM and agent logs (`fake_vlm_q`, `fake_scratchpad`) are placeholders that exercise the database write path without needing real models. They also demonstrate the format your real `vlm.py` and `agent.py` will need to produce.

One thing to note: the scripted moves stay in the top-left corner of the grid, which is far from the camera in the top-right. That's why the smoke test output shows `hidden ✓` for every step — the agent never wanders into camera range. That's fine for a smoke test, but for Day 2 you'll want to verify the loop actually hits `VISIBLE ⚠` at some point during a real run.
